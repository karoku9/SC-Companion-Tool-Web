# v0.24 — OCR Assisted Intake

## Purpose

OCR is a secondary mission-intake path for screenshots and cropped contract images. It does not replace manual text or Game.log intake. Its job is to produce a reviewable draft while keeping recognition uncertainty and image-to-field provenance visible.

## User flow

1. Select one to six contract images explicitly in **Missions**.
2. The browser preprocesses each image and starts the pinned OCR worker.
3. Review the image preview, raw recognized text and extracted fields.
4. Correct title, action, destination, SCU or commodity inline.
5. Load the OCR draft into the canonical mission review.
6. Resolve every validation blocker.
7. Generate the session explicitly.

Image selection, recognition and draft loading never replace the active route.

## OCR runtime

- Browser engine: Tesseract.js 7.0.0.
- Module source: exact-version ESM URL on jsDelivr.
- Language: English.
- Page segmentation: sparse text.
- Worker creation is lazy and reused for the current page session.
- The engine can be cancelled and terminated.
- The runtime, WebAssembly core and English language model require network retrieval on first use.
- Offline availability after first use depends on browser cache behaviour and is not guaranteed by the application.

## Privacy and persistence

- Application code does not send selected images to an application server.
- Image pixels are processed by the browser worker.
- Source image bytes and preview object URLs are not stored in session state.
- Stored state is bounded to recent sanitized OCR reports.
- A report may retain extracted text, source filename, type, size, hash, processed dimensions, engine/version, field confidence and OCR-line provenance.
- Restored reports show that the source preview is unavailable until the user selects the image again.

## Preprocessing

- Decode selected images in the browser.
- Bound the longest processed dimension to 2,400 pixels.
- Enlarge small images by at most two times.
- Draw onto an opaque canvas.
- Apply grayscale and increased contrast.
- Sample luminance and invert dark HUD screenshots automatically.
- Preserve the original file preview for visual comparison during the active page session.

Preprocessing is a recognition aid, not a source of verified contract data.

## Field extraction

`ocr-intake.js` receives recognized text plus optional per-line confidence and produces:

- mission title;
- objective action;
- operational destination;
- positive SCU quantity;
- commodity;
- per-field confidence;
- per-field source line and extraction method;
- complete or partial objective status.

Explicit action headings are authoritative anchors. Derived `From:` and `To:` labels are used only when no explicit action headings exist. This prevents destination rows and titles containing “delivery” from becoming duplicate objectives.

Destinations resolve through the canonical operational location registry. Ambiguous and unresolved locations remain explicit. Missing quantity or commodity stays missing rather than being synthesized.

## Review boundary

OCR serialization produces the same compact mission-text format used by manual intake:

```text
Mission title
collect ARC-L2 Lively Pathway Station 3scu titanium
deliver Teasa Spaceport 3scu titanium
```

The existing mission validator remains authoritative for:

- supported actions;
- known, ambiguous and custom locations;
- positive SCU values;
- commodity syntax;
- complete pickup-to-delivery flow;
- stale review after edits;
- explicit session generation.

## Verification

Node contracts cover:

- independent field extraction;
- incomplete-field preservation;
- explicit correction serialization;
- explicit versus derived action anchors;
- source metadata and line provenance;
- pinned OCR dependency and review-first runtime boundary.

Chromium coverage performs:

- real browser image upload;
- browser image decoding and preprocessing;
- deterministic mocked Tesseract ESM loading;
- two-objective field rendering;
- OCR-line provenance display;
- editable field interaction;
- draft handoff to a ready mission review;
- proof that route state remains null before explicit generation;
- desktop and 390-pixel mobile overflow checks.

The main quality workflow uses `set -o pipefail`, so failures in additional test suites cannot be hidden by log piping.

## Known limits

- OCR quality depends on crop, scale, font rendering, compression, contrast and in-game UI layout.
- Only English recognition is configured.
- Multiple screenshots are treated as separate mission reports rather than automatically stitched pages.
- The parser supports common action/location/quantity/commodity layouts but does not claim universal coverage of every Star Citizen contract screen.
- Real gameplay screenshots remain necessary to tune preprocessing and layout patterns empirically.
